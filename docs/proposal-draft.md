# Think-Prompt (가칭) — 서비스 초안

> 본 문서는 브레인스토밍/논의를 바탕으로 정리한 **초안(draft)** 이다.
> 1차 의사결정은 §0.5에 확정 기록. 이후 결정은 동일 위치에 append.

---

## 0. 한 줄 요약

**Claude Code를 시작점으로**, 개발자가 실제로 입력하는 프롬프트를 **로컬 중심으로 수집·진단·개선**해 주는 **개인 개발자용 프롬프트 코치**.

단순 로깅이 아니라 **수집 + 개선을 동시에** 제공. 무료로 공개해 사용자 풀을 만든 뒤, B2B/팀 기능은 추후 판단.

| # | 결정 항목 | 선택 | 영향 |
|---|---|---|---|

| 6 | 품질 스코어 | 룰베이스 70% + 실사용 30%, LLM 심판은 **의심 케이스 trigger**로만 | 초기 LLM 비용 절감, 룰셋 정의가 최우선 작업 |
| 7 | 수익 모델 | **현재 논의 대상 아님** (무료) | seat/플랜 설계 없음 |

**결정이 기존 초안을 바꾸는 지점 요약:**
- §3(수집 레이어): P0는 **Claude Code 훅 단독**. 서버 SDK·CLI 래퍼는 Phase 2로 후순위.
- §6(프라이버시): 기본 모드는 **로컬 우선(B)**. 서버는 선택적 기능.
- §7(UX): 팀 대시보드·PR 워크플로 → **개인 대시보드**로 축소. Prompt Editor는 diff만 유지.
- §8(로드맵): Phase 2/3의 팀 협업·RBAC·PR은 **보류 목록**으로 이동.

## 1. 기존 초안 요약

여러 플랫폼(Claude Code / ChatGPT / Gemini / Cursor / CLI / 사내 챗봇 등)에서 **유저가 실제로 입력하는 프롬프트를 통합 수집·관리하고**, 품질을 진단해 **“어디가 잘못됐고 어떻게 고치면 좋은지”를 코치처럼 안내**하는 서비스.

단순 로깅/모니터링이 아니라 **프롬프트 개선(리팩터링/리라이팅/A/B)을 1급 기능**으로 삼는다는 점이 차별점.

> **※ §0.5 결정에 따라 MVP는 위 범위의 부분집합(Claude Code + 개인 + 로컬)으로 시작한다.**

---

## 1. 목표와 기본 원칙

### 1.1 목표
- 여러 툴에서 흩어져 있는 프롬프트를 **한 레포지토리**에 모은다.
- 모인 프롬프트를 **품질 기준으로 진단**한다.
- **자동 리라이트/템플릿 추천/A/B 테스트**로 개선까지 이어지는 워크플로를 만든다.
- 팀 단위로 프롬프트를 **코드처럼 PR·리뷰·배포**할 수 있게 한다.

### 1.2 기본 원칙
- **프라이버시 우선(Privacy by Design):** 프롬프트에는 개인정보·기밀이 섞이므로 Opt-in / 마스킹 / 보존 정책을 1급 요소로 다룬다.
- **감시가 아니라 코칭:** “Always-on 감시”가 아닌 “Always-on 코칭”. 기본은 집계 수준, 상세 로그는 유저가 명시적으로 켤 때만.
- **1클릭 개선:** 진단만 하고 끝내지 않는다. 모든 지적 옆에는 “이 버튼으로 고친다”가 있어야 한다.
- **개발자 + 실무자 혼합 타겟:** API 호출뿐 아니라 IDE/CLI/챗 UI에서 사람이 치는 프롬프트까지 포괄.

## 3. 수집 레이어 — 채널별 가능성 정리

| 채널 | 방식 | 기술적 가능성 | 주요 리스크 | 초기 우선순위 |
|---|---|---|---|---|
| 백엔드/서버 LLM 호출 | 자체 SDK(Proxy)로 OpenAI/Anthropic/Gemini 호출 감싸기 | 높음 — 업계 표준 패턴 | 낮음 (사내 적용) | **P0** |
| CLI (claude / gemini-cli 등) | 래퍼 바이너리 + 로컬 에이전트(daemon), alias 또는 `LLM_PROXY_ENDPOINT` | 높음 — 구현 난이도 낮음 | “모든 CLI 감시” 느낌, 신뢰 이슈 | **P0** |
| Claude Code / Harness | LLM 호출 훅, 서브에이전트별 시스템 프롬프트 추출 | 높음 — 이미 hook 포인트 존재 | 회사 환경 종속성 | **P0** |
| Cursor / VSCode / JetBrains | 확장(Extension) API로 LLM 패널 입출력 후킹 | 중간 — 각 IDE별 API 상이 | 스토어 정책, UX 마찰 | P1 |
| 브라우저 웹 UI (ChatGPT/Claude/Gemini) | Chrome/Edge 확장으로 입력창 DOM 읽기 | 가능하나 민감 | **악성 확장과 동일 메커니즘** — 신뢰·스토어 정책 | P2 |
| 사내 챗봇/커스텀 웹툴 | 프록시 엔드포인트 경유 로깅 | 높음 | 낮음 | P1 |

### 3.1 CLI / 터미널 훅 구조(현실적 설계)

- **로컬 에이전트 (daemon):** `think-prompt-agent` 가 `localhost:port`에 상주. CLI 래퍼 → 에이전트 POST → PII 마스킹 → 로컬 DB 또는 중앙 서버 전송.
- **래퍼 방식 3종:**
  - 쉘 alias: `alias claude='think-prompt-cli claude'`
  - wrapper binary: `pro-claude` 가 stdin/stdout을 중계
  - OTel/HTTP 프록시: 일부 툴은 이 방식을 공식 지원
- **장점:** 구현 난이도 낮고, Observability 업계의 기존 패턴과 정확히 일치.
- **리스크:** “키로거처럼 보일 수 있음” — 설치 시 “어떤 명령의 무엇을 캡처하는지” 명시적 UI 필요.

### 3.2 브라우저 확장 — 가능하지만 민감

- 기술적으로는 `ChatGPT/Claude/Gemini` 웹의 입력창 DOM을 읽는 확장 구현이 완전히 가능하며, 유사 선행 사례(Lakera, GPT Privacy 등)도 다수 존재.
- 그러나 **악성 확장이 동일 메커니즘으로 프롬프트를 외부 서버로 유출한 사례**(Urban VPN 계열 등)가 보고되어 있어, 설계가 잘못되면 “키로거 / 스파이웨어”로 분류될 수 있음.
- **완화 방안:**
  - **On-device 우선:** PII 탐지·마스킹은 브라우저 안에서. 외부로는 최소 메타데이터만.
  - **Scope 명시:** `chatgpt.com`, `claude.ai`, `gemini.google.com` 등으로 host_permissions를 한정하고, 작동 사이트를 항상 UI로 표시.
  - **세분화된 Opt-in:** 카테고리 토글(개인 메일/계약/의료/금융 → 기본 off).
  - **투명 정책:** 어떤 데이터를 어디로 보내고 언제 지우는지 스토어 설명과 앱 내 정책에 동일하게 기술.

### 결정 필요
- [ ] MVP에서 지원할 채널 범위(추천: **백엔드 SDK + CLI 래퍼 + Claude Code 훅**만 P0로 시작)
- [ ] 브라우저 확장은 별도 트랙(P2)로 두고, 먼저 “수동 공유 버튼”으로 대체 가능한지 검증

## 4. 저장 · 모델링 레이어

### 4.1 코어 엔티티

| 엔티티 | 주요 필드 | 용도 |
|---|---|---|
| `Prompt` | id, version, hash, type(system/user/tool), tags, owner, created_at, last_used_at | 프롬프트 자체. Git repo 유사 구조. |
| `PromptUsage` | prompt_id, version, model, env(prod/stg), user, input_size, output_size, latency, error | 실 사용 이벤트. |
| `Outcome` / `Feedback` | usage_id, rating(👍/👎), NPS, flags(유용/헛소리/위험), freeform | 결과 품질. |
| `Version` | prompt_id, parent_version, diff, branch, status(draft/active/archived) | 버전·브랜치. |
| `Tag` | name, scope(task/domain/product) | 검색·클러스터링. |

### 4.2 버전 관리
- `promptId + version`을 키로 보관, **브랜치/머지/롤백** 개념 지원.
- 한 프롬프트에 **실험 브랜치**를 만들고, A/B에서 이기면 `main`으로 승격.

## 5. 분석 · 평가 · 개선 레이어

### 5.1 진단(Diagnostics) — 뭐가 문제인지 자동으로 찾기

**룰베이스 + LLM 메타평가 혼합**으로 `Prompt Quality Score` 산출.

**안티패턴 체크리스트(룰베이스):**
- 과도하게 짧거나 모호한 요청 (“summarize please” 수준)
- 복수 태스크 혼합 (요약+번역+코드를 한 번에)
- 컨텍스트 부족 (도메인/대상/목적 누락)
- 출력 포맷 미지정 (JSON/표/길이 등)
- 프롬프트 인젝션 유도 패턴 (“ignore previous instructions” 등)

**품질 메트릭(사용 데이터):**
- 성공률 / 에러율
- 포맷 적중률 (요구 JSON 파싱 실패율 등)
- 유저 피드백 평균 (👍/👎)
- 후속 재질문(리트라이) 횟수
- 응답 길이, 지연시간

### 5.2 코칭(Coaching) — 고치는 방법까지 같이 주기

- **즉석 인라인 피드백:** IDE/CLI 입력 UX에 회색 힌트로 “컨텍스트가 부족합니다…” 바로 표시.
- **자동 리라이팅:** 프롬프트를 LLM에 넘겨 **Instruction / Context / Constraints / Output Format / Examples** 포맷으로 재구성한 버전 제안, 1클릭 채택.
- **템플릿 추천:** 태스크 유형(요약/번역/코드리뷰 등) 분류 후 해당 템플릿 자동 매핑.
- **A/B 테스트:** 동일 엔드포인트에 기존 vs 개선안을 50/50 또는 20/80으로 배포 → 성공률·피드백 비교 → 우승안을 `main`으로 자동 승격.
- **모델별 튜닝 노트:** 같은 베이스 프롬프트로부터 Claude / ChatGPT / Gemini 각각에 맞는 문구 조정 제안.

### 5.3 개선 전/후 비교 카드

- 유저가 주관적으로 받아들이지 않도록 **숫자로 증명**:
  - “기존 프롬프트: 성공률 62%, 평균 재질문 3.2회 → 개선 프롬프트: 성공률 84%, 재질문 1.1회”

## 6. 프라이버시 · 보안 · 동의 구조 (★ 1급 요소)

프롬프트는 개인정보·사내 기밀이 섞이기 쉬우므로 **설계 초기부터** 아래 요소를 고정한다.

### 6.1 동의(Opt-in / Opt-out)
- 워크스페이스 단위 + 유저 단위 2단 옵션:
  - 원문 전체 저장 허용
  - 민감정보 제거 후 저장 (기본값)
  - 저장 안 함 (메트릭만)

### 6.2 PII / 민감정보 처리
- 이메일 / 전화번호 / 주민번호 / 계좌 / 사람명·주소 등을 **on-device 또는 edge**에서 토큰화·마스킹.
- 브라우저 확장·CLI 에이전트 모두 **원문은 가급적 로컬에서 처리**하고 서버에는 마스킹된 결과만 전송.

### 6.3 보관 정책
- 원문 보관 기간(기본 30일 / 90일 / 무기한 중 선택)
- 일정 기간 후 요약/통계만 남기고 원문 삭제
- 데이터 주체 삭제 요청 처리 플로우

### 6.4 정책 투명성
- “우리는 프롬프트 데이터로 **무엇을 하고 무엇은 절대 하지 않는가**”를 약관·스토어 설명·앱 내 설정에 **동일하게** 기재.

### 결정 필요
- [ ] PII 탐지를 자체 룰 + 오픈소스(PresidIO 등) vs 전용 모델 중 어디로
- [ ] 저장 기본값: 마스킹 후 서버 저장 vs 원문 로컬 저장 + 서버는 해시만
- [ ] 데이터 지역성 요건(한국/EU 고객 대응)

## 7. UX / UI

### 7.1 대시보드
- LLM별 / 서비스별 / 팀별 **프롬프트 목록 + 성능 테이블**
- “최근 7일 최다 사용 Top N”, “품질 낮은 프롬프트 Top N”
- “문제 프롬프트 랭킹” — 각 항목에 원인 아이콘(모호 / 컨텍스트 부족 / 포맷 없음)

### 7.2 Prompt Editor
- 좌: 이전 버전 / 우: 새 버전 diff / 하: 메트릭 비교 그래프
- 리뷰 코멘트, PR 승인, 배포 토글

### 7.3 IDE / CLI 인라인 UI

### Phase 1 · MVP (로깅 + 뷰어 + 간단 개선)

## 9. 경쟁 지형 & 차별화

| 기존 툴 영역 | 특징 | 이 서비스가 가져갈 포인트 |
|---|---|---|
| PromptLayer 계열 | API 호출 로깅·히스토리·검색 | SDK/Proxy 방식은 그대로 차용, **CLI·IDE·챗UI까지 수집 범위 확장** |
| PromptRepository 프레임워크 | JSON 기반 ID/버전, 템플릿 파라미터, 타입세이프 | **프로덕션 프롬프트**를 템플릿/타입으로 정규화해 재사용 |
| prst.ai 등 프롬프트 관리 툴 | 다중 LLM, A/B, 피드백, 벡터DB 연동 | 다중 모델 라우팅 + 지식베이스 결합(고급 플랜) |

**차별화 핵심:**
1. **개발자 + 실무자 혼합 타겟** — API뿐 아니라 IDE/CLI/챗 UI까지 한 곳에서.
2. **“개선”을 1급 기능으로** — 단순 로깅이 아니라 자동 리팩터링 · 모델별 튜닝 · A/B 승격이 메인.
3. **Git-like 프롬프트 레포지토리** — PR·리뷰·승인·배포.
